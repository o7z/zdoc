# 方案 A：5 层建设者视角

适合**正在被构建**的、有多个组件的软件系统。文档的读者是**造系统的人**——开发者、架构师、AI 协作者。

## 目录布局

```
docs/
├── 0-glossary/      术语：项目说什么语言
├── 1-vision/        为什么做、做成什么
├── 2-architecture/  有哪些组件、怎么协作
├── 3-contract/      组件间怎么握手
├── 4-design/        每个组件怎么实现
├── 5-ops/           部署与运维
├── references/      外部参考项目（旁路）
└── testing/         测试场景日志（旁路）
```

## 逻辑

按一个系统**被构建**的自然顺序分层。每一层回答一个明确的问题，后一层基于前一层的决定。

- `0-glossary/` 是所有后续文档的**语言地基**——先约定"我们说什么话"，后面每一层才不会各说各话
- 读者可以按 0 → 5 的顺序渐进理解项目
- AI 协作者能用同一条路径快速爬坡（读 glossary 对齐术语 → 读 vision 对齐意图 → 读 architecture 知道拼图 → 读 contract 知道边界 → 改 design → 看 ops）
- 数字前缀同时表达"顺序"和"依赖"：改 3-contract 要看懂 2-architecture；改 4-design 要看懂 3-contract
- 非数字目录（`references/`、`testing/`）是服务主干的**旁路资产**，不参与主干编号

## 适合

- 后端服务、Agent 平台、多服务项目
- 自研框架 / 平台 / SDK 的核心仓库
- AI 驱动协作的长期项目（结构稳定 → AI 不会每次自由发挥）

## 不适合

- 单文件 CLI、纯前端 App —— `3-contract/` 会空掉
- 文档内容主要是"用户怎么用"（而不是"系统怎么造") —— 用[方案 B](./structure-user)
- 知识零散、主题不聚焦 —— 用[方案 C](./structure-knowledge)
- 团队极小且术语无歧义 —— `0-glossary/` 可省略

## 0-glossary/ 的两类术语

术语文档建议区分两类，它们服务不同读者、解决不同问题：

| 类型 | 解决什么问题 | 举例 |
|------|-------------|------|
| **领域术语** | 业务 / 产品概念的统一理解，避免跨角色沟通时"同一个词各理解各的" | "订单"在电商系统里是已支付还是已提交、"租户"指组织还是空间 |
| **开发专用术语** | 开发期团队内部交流避免歧义的特定短语——**必须避免用常用英文单词造成理解歧义** | 用 `reconcile-loop` 而非 `sync`、用 `warm-restart` 而非 `reload`、用 `fan-out-write` 而非 `broadcast` |

开发专用术语的核心原则：**如果一个常用英文单词在项目上下文中可能被理解为多种行为，就必须发明一个无歧义短语替代它。** 这对 AI 协作尤其关键——AI 不会从语境推断你的特指含义，它只会按最常见的字典义理解。

## 可裁剪

**简单项目可直接省略 `3-contract/`**，合并到 `2-architecture/` 或 `4-design/` 里。数字编号不需要连续。

## AGENTS.md 建议

如果使用方案 A，强烈建议在项目根目录放置 `AGENTS.md`，让 AI 协作者（Cursor、Claude Code、Copilot 等）自动理解项目文档结构。以下是一个起点：

```markdown
# AGENTS.md

## 文档结构

本项目使用 zdoc 管理文档，采用 5 层建设者视角分层：

- `0-glossary/` — 术语定义（阅读任何文档前先看这里）
- `1-vision/` — 项目愿景与目标
- `2-architecture/` — 系统架构与组件关系
- `3-contract/` — 组件间接口契约
- `4-design/` — 各组件实现设计
- `5-ops/` — 部署与运维

## 协作规则

- 改代码前先读对应层级的文档；改文档时遵循分层依赖（改 4-design 要看懂 3-contract）
- 新增概念必须先在 `0-glossary/` 定义术语
- 接口变更必须同步更新 `3-contract/`
```

按项目实际情况增减即可。`AGENTS.md` 的核心价值是让 AI 在每次会话开始时**零成本**理解"文档在哪里、怎么读、改代码时该看什么"。
