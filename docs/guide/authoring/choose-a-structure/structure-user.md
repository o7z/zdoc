# 方案 B：Diátaxis 四象限

适合**已稳定、面向终端用户**的产品文档。文档的读者是**用系统的人**——终端用户、SDK 使用者、API 调用者。

[Diátaxis](https://diataxis.fr/) 是 Django / NumPy / Cloudflare 等大项目采用的现成理论。按**读者此刻的意图**分类，不是按信息来源。

## 目录布局

```
docs/
├── tutorials/     教程（学习向）
├── how-to/        指南（任务向）
├── reference/     参考（事实向）
└── explanation/   说明（背景向）
```

## 逻辑

按**读者此刻的意图**分类，不是按信息来源：

| 目录         | 读者在做什么      | 读者需要什么          |
|--------------|-------------------|-----------------------|
| tutorials    | 我想学一门新东西  | 被牵着手走完全流程    |
| how-to       | 我要完成一个任务  | 精准配方，不要废话    |
| reference    | 我要查一个事实    | 准确、完整、可 grep   |
| explanation  | 我想懂原理        | 背景、权衡、设计动机  |

四象限边界清楚，新文档往往一眼就知道归哪类；归类困难的文档往往是因为它**同时在做两件事**，这种时候拆成两篇反而更好读。

## 适合

- 已发布的 SDK、API、库
- 面向终端用户的产品（含 SaaS）
- 文档的主体读者是"用这东西的人"，不是"造这东西的人"

## 不适合

- 还在搭骨架、边写边改的系统 —— 用[方案 A](./structure-builder.md)
- 主要是团队内部的决策记录、ADR 集合 —— 这些在 Diátaxis 下会被硬塞进 explanation，挤压到失真

## AGENTS.md 建议

如果使用方案 B，建议在项目根目录放置 `AGENTS.md`，帮助 AI 协作者理解文档的四象限划分：

```markdown
# AGENTS.md

## 文档结构

本项目使用 zdoc 管理文档，采用 Diátaxis 四象限结构：

- `tutorials/` — 教程：引导新手走完一个完整流程
- `how-to/` — 指南：面向任务的精准配方
- `reference/` — 参考：API、配置项、事实性信息
- `explanation/` — 说明：设计动机、架构背景、权衡取舍

## 协作规则

- 新文档先判断读者意图，再决定放哪个象限
- 一篇文档只做一件事：教、导、查、释，不要混搭
- reference 文档保持客观事实，不加教程式引导
```
