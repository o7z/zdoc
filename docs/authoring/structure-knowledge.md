# 方案 C：扁平 + 搜索

适合**知识零散、没有明显主干**的内容库。文档的读者是**查资料的人**——运维工程师、笔记读者、知识库浏览者。

## 目录布局

```
docs/
├── _meta.yaml      # 所有 .md 全列在 pages 下，靠 order 排序
├── index.md
├── setting-up-ssh.md
├── deploying-to-staging.md
├── debugging-the-build-cache.md
└── ...
```

或者一级简分类 + 其下扁平，不再更深：

```
docs/
├── ops/            （扁平，不再细分）
├── dev/            （扁平，不再细分）
└── misc/
```

## 逻辑

层级在内容增长时会**腐烂** —— 每新写一篇文章都要决策归哪层，而分层一旦定死就很难重构。扁平结构把"在哪里"的问题转移给搜索和标签，成本低很多。

- zdoc 自带 `Ctrl+K` 模糊搜索，扁平结构仍然可导航
- 文档不断增长时只是列表变长，不会出现"我要建一个四级子目录才装得下这个"
- 更符合"每篇文档独立可读"的原则，不依赖上下文页

## 适合

- 个人 Zettelkasten / 笔记系统
- 团队运维手册（主题跳跃，强时序）
- 给客户的"各种小知识合集"

## 不适合

- 需要读者按固定顺序读完的内容 —— 用[方案 A](./structure-builder) 的渐进路径
- 文档量小（< 30 篇）—— 随便分都不会腐烂，[方案 B](./structure-user) 更像文档

## AGENTS.md 建议

如果使用方案 C，建议在项目根目录放置 `AGENTS.md`，帮助 AI 协作者理解扁平结构的导航方式：

```markdown
# AGENTS.md

## 文档结构

本项目使用 zdoc 管理文档，采用扁平 + 搜索结构：

- 文档不按层级组织，靠 `_meta.yaml` 中的 order 排序
- 使用 `Ctrl+K` 搜索定位文档
- 每篇文档独立可读，不依赖上下文页

## 协作规则

- 新文档直接加到根目录或一级子目录，不要创建更深层级
- 文档标题要自描述，方便搜索命中
- 如果发现某主题文档超过 10 篇，可以考虑提取为一级子目录
```
