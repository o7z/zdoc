# 选一套文档结构

> **zdoc 不强制任何结构。** 只要 `_meta.yaml` 合法，任何分层方式都能被渲染。
>
> 但"哪个文档放哪里"往往是起步时最耗心力的决定。下面列出三套经过实践验证的方案，以及它们各自的逻辑与适用范围。**挑一个、改一个、或者完全忽略都行。**

## 方案 A：5 层建设者视角

适合**正在被构建**的、有多个组件的软件系统。

### 目录布局

```
docs/
├── 1-vision/        为什么做、做成什么
├── 2-architecture/  有哪些组件、怎么协作
├── 3-contract/      组件间怎么握手
├── 4-design/        每个组件怎么实现
├── 5-ops/           部署与运维
├── references/      外部参考项目（旁路）
└── testing/         测试场景日志（旁路）
```

### 逻辑

按一个系统**被构建**的自然顺序分层。每一层回答一个明确的问题，后一层基于前一层的决定。

- 读者可以按 1 → 5 的顺序渐进理解项目
- AI 协作者能用同一条路径快速爬坡（读 vision 对齐意图 → 读 architecture 知道拼图 → 读 contract 知道边界 → 改 design → 看 ops）
- 数字前缀同时表达"顺序"和"依赖"：改 3-contract 要看懂 2-architecture；改 4-design 要看懂 3-contract
- 非数字目录（`references/`、`testing/`）是服务主干的**旁路资产**，不参与主干编号

### 适合

- 后端服务、Agent 平台、多服务项目
- 自研框架 / 平台 / SDK 的核心仓库
- AI 驱动协作的长期项目（结构稳定 → AI 不会每次自由发挥）

### 不适合

- 单文件 CLI、纯前端 App —— `3-contract/` 会空掉
- 文档内容主要是"用户怎么用"（而不是"系统怎么造") —— 用方案 B
- 知识零散、主题不聚焦 —— 用方案 C

### 可裁剪

**简单项目可直接省略 `3-contract/`**，合并到 `2-architecture/` 或 `4-design/` 里。数字编号不需要连续。

## 方案 B：Diátaxis 四象限

适合**已稳定、面向终端用户**的产品文档。[Diátaxis](https://diataxis.fr/) 是 Django / NumPy / Cloudflare 等大项目采用的现成理论。

### 目录布局

```
docs/
├── tutorials/     教程（学习向）
├── how-to/        指南（任务向）
├── reference/     参考（事实向）
└── explanation/   说明（背景向）
```

### 逻辑

按**读者此刻的意图**分类，不是按信息来源：

| 目录         | 读者在做什么      | 读者需要什么          |
|--------------|-------------------|-----------------------|
| tutorials    | 我想学一门新东西  | 被牵着手走完全流程    |
| how-to       | 我要完成一个任务  | 精准配方，不要废话    |
| reference    | 我要查一个事实    | 准确、完整、可 grep   |
| explanation  | 我想懂原理        | 背景、权衡、设计动机  |

四象限边界清楚，新文档往往一眼就知道归哪类；归类困难的文档往往是因为它**同时在做两件事**，这种时候拆成两篇反而更好读。

### 适合

- 已发布的 SDK、API、库
- 面向终端用户的产品（含 SaaS）
- 文档的主体读者是"用这东西的人"，不是"造这东西的人"

### 不适合

- 还在搭骨架、边写边改的系统 —— 用方案 A
- 主要是团队内部的决策记录、ADR 集合 —— 这些在 Diátaxis 下会被硬塞进 explanation，挤压到失真

## 方案 C：扁平 + 搜索

适合**知识零散、没有明显主干**的内容库：个人笔记、团队 wiki、碎片化资料。

### 目录布局

```
docs/
├── _meta.yaml      # 所有 .md 全列在 pages 下，靠 order 排序
├── index.md
├── setting-up-ssh.md
├── deploying-to-staging.md
├── debugging-the-build-cache.md
└── ...
```

或者一级简分类 + 其下扁平，不再更深：

```
docs/
├── ops/            （扁平，不再细分）
├── dev/            （扁平，不再细分）
└── misc/
```

### 逻辑

层级在内容增长时会**腐烂** —— 每新写一篇文章都要决策归哪层，而分层一旦定死就很难重构。扁平结构把"在哪里"的问题转移给搜索和标签，成本低很多。

- zdoc 自带 `Ctrl+K` 模糊搜索，扁平结构仍然可导航
- 文档不断增长时只是列表变长，不会出现"我要建一个四级子目录才装得下这个"
- 更符合"每篇文档独立可读"的原则，不依赖上下文页

### 适合

- 个人 Zettelkasten / 笔记系统
- 团队运维手册（主题跳跃，强时序）
- 给客户的"各种小知识合集"

### 不适合

- 需要读者按固定顺序读完的内容 —— 用方案 A 的渐进路径
- 文档量小（< 30 篇）—— 随便分都不会腐烂，方案 B 更像文档

---

## 怎么选

一个三分支决策规则：

| 你的处境                                      | 用哪个 |
|-----------------------------------------------|--------|
| 系统正在搭建，希望 AI 也能理解并持续协作      | **A**  |
| 产品已稳定，文档读者是"用这东西的人"          | **B**  |
| 内容多且零散，更像知识库而非产品手册          | **C**  |

边缘情况：

- **A + 后期迁 B**：产品在 A 结构下做完 1-vision / 2-architecture 稳定后，对外输出的文档可以以 Diátaxis 再做一层——两者可以共存在同一仓库的不同子树
- **B + 标签**：标签可以跨 Diátaxis 四象限打横索引（比如 "authentication" 标签同时收 tutorials/auth-login 和 reference/jwt-claims）
- **C + 加一级**：内容超过 ~100 篇时通常会浮现几个天然主题，那时给扁平再加一级是自然而然的

## 可以自己发明

以上没有一个是"官方标准"。如果你脑中已经有另一种更合适的组织方式，**就用你的**。`_meta.yaml` 只关心结构是否合法，对语义毫无要求。zdoc 能渲染的结构，都是好结构。
